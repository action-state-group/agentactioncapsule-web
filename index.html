A verifiable record of what an AI agent did.

An Agent Action Capsule is a tamper-evident, content-private, independently verifiable record of what an AI agent did — checkable by anyone, without trusting the operator or any single layer. Supply-chain attestation, applied to agent actions instead of build artifacts.

The TLS of agent actions: the verifiable record is the headline; the cryptography is just how it holds.

The problem

Agents take actions. Right now, you have to trust the operator that they happened as claimed.

The Agent Action Capsule closes that gap with three properties — the same shape SCITT defines for software supply chains, profiled for agent actions.

Signed

Sealed at the moment of action

A capsule is a COSE-signed statement committing to the action, its inputs and outputs (by digest), and the model/runtime that produced it. Tamper any byte and verification fails.

Transparent

Anchored to an append-only log

The statement is registered to a SCITT Transparency Service, which returns a receipt proving inclusion. The log can't quietly drop or rewrite history.

Third-party verifiable

Anyone can check, offline

An auditor, reviewer, or regulator verifies the signature and the inclusion proof from the bytes alone — no access to the operator's systems, no trust required.

What's sealed

Purpose-built for agent actions — not generic attestation.

The seal is the capsule_id — a SHA-256 of the whole statement; recompute it and it must match, byte for byte. And it's content-private by construction: a capsule carries digests of your inputs and outputs, never the raw prompts, vendors, or amounts. Around that seal it commits exactly what an agent action needs to be accountable:

Decision

The may/did, bound to its effect

What the agent decided and whether it executed, was confirmed, denied, or blocked — with a confirmed-effect binding, so the claim and the real outcome can't drift apart.

Attestation

Which model decided, by digest

The model and runtime that produced the action, with its input and output committed as SHA-256 digests. The proof travels; the raw values stay with you.

Chain

Trails for human-in-the-loop

Confirm, supersede, and escalate links turn approved → executed → confirmed into one verifiable chain — the basis for review and selective disclosure.

That's the shape a build-artifact attestation doesn't give you — and the reason to standardize one agent-action profile together rather than reinvent it. What's in a capsule →

The stack

One statement layer. Any transparency service. Neutral, donatable substrate.

The capsule is a statement-layer profile — it makes no claim about which verifiable-data-structure the log uses. Each layer is a separate open-source library.

layer 1The profile
agent-action-capsule — the Signed Statement format (COSE_Sign1 over application/agent-action-capsule+json) + the reference verifier.
↓ produced by
layer 2The producer
capsule-emit — seal an action in one call, or wrap an existing tool with one decorator. The agent-developer on-ramp.
↓ registered to · receipt verified by
layer 3The verifier
scitt-cose — verifies SCITT receipts: RFC9162_SHA256 (vds=1) and CCF ccf.v1 (vds=2).
↓ anchored to
layer 4The log
capsule-anchor — a neutral, product-free SCITT Transparency Service. Live at anchor.agentactioncapsule.org.
Verify

From the bytes. Offline. No network, no trust.

This is what an auditor runs against a ledger of capsules. Tamper any field and ok turns false.

# install the reference verifier
pip install agent-action-capsule

# verify a ledger of sealed actions, offline
agent-action-capsule verify --store ledger.jsonl

  capsule_id  9f2a…c14  ok
  capsule_id  4b07…8de  ok
  # inclusion on the public log:
  substrate.receipt_verified: True
1

Inputs & outputs committed by digest

The capsule seals SHA-256 digests of what went in and came out. Raw values stay local; the proof travels.

2

Signature + inclusion, independently

The signature proves who sealed it; the receipt proves the log included it. Both check from the bytes.

3

VDS-agnostic

The same statement verifies whether the log uses RFC 9162 Merkle proofs or Microsoft CCF. The action layer never changes.

The on-ramp

Any MCP tool call → a verifiable record, in one decorator.

No protocol change. The capsule is the record layer you compose into your existing agent or MCP server — the input and output are sealed automatically.

# wrap a tool — capsule-emit seals every call
@emitter.tool(effect_type="write_order")
def submit_order(vendor, amount, po_number):
    # your real tool logic
    return place(vendor, amount, po_number)

# call it normally — a sealed, anchored
# record is written for every invocation
submit_order("Frobozz", 4210.00, "PO-0047")
✓

Drop-in

One decorator over a function or MCP tool. The signature is preserved, so your MCP schema is unchanged.

✓

Auto-anchored

Each call seals input + output digests and fires the statement to the transparency log. Verifiable immediately.

✓

Adapter, not lock-in

MCP today; the same producer composes into other agent runtimes. The substrate is neutral and donatable.

Adapters

Seal from the framework you already use.

One capsule, one anchor, one verify — an adapter only changes where you hook in. Each is a thin layer over a ~30-line base, so adding your own is small. The format spreads by being the easiest thing to emit into — so integrators put it everywhere.

DECORATOR

MCP / any callable

A decorator on the tool function — the input args and return value are digested automatically.

CALLBACK

LangChain / LangGraph

A callback handler on the run; the model id auto-fills where the framework exposes it.

TOOL WRAP

CrewAI

Wraps the tool object — one capsule per tool call, no per-call-site code.

MCP SERVER

Goose

A companion MCP server exposing record · verify · ledger to a Goose session.

ANY LOOP

Hermes / custom loop

One call at your tool boundary — the "any framework" pattern on the shared base.

YOURS?

Don't see your framework?

The base is ~30 lines. Add an adapter, or open a request — capsule producers stay one shared format.
